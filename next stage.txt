Next Stage Plan (updated: 2026-03-02)

方向
- Research first：优先做机制效率与推理质量研究，不先追求大而全产品。
- 产品形态：Flatter + React Web App（输入 API Key、PDF/LaTeX、目标定理）。
- 核心指标：完成度、token 消耗、时间消耗、cost、可复现性。

统一评估指标（所有阶段共用）
- Success rate：定理通过率（`lake build` + sorry 消除）。
- Token / theorem：每个定理平均 token。
- Time / theorem：每个定理平均端到端耗时。
- Cost / theorem：按模型实际价格核算。
- Pass@k：同一目标多次尝试成功率。

第一阶段（2026-03-02 ~ 2026-04-30）：Benchmark + 数据闭环
目标
- 系统比较闭源/开源模型在形式化证明任务上的效果、效率、成本。
- 建立可复用评测集与日志规范，为后续微调准备数据。

任务
- 选模型（闭源 + 开源旗舰 + 开源中型）并统一 prompt/template。
- 固定 benchmark 集（按难度分层：easy/medium/hard）。
- 跑批并记录：成功率、token、耗时、成本、失败类型。
- 做 token 优化实验（prompt 压缩、检索裁剪、分阶段调用）。
- 设计用户数据授权流程：用户可同意将脱敏交互数据 commit 到指定仓库。

交付物
- benchmark 报告（模型完成度/cost/效率对比表）。
- 数据 schema（任务输入、推理轨迹、编译错误、最终结果）。
- PRD v1（与老师讨论版本）。

验收
- 至少形成一版可复现实验脚本与结果表。
- 明确下一阶段首选模型组合与预算上限。

第二阶段（2026-05-01 ~ 2026-07-31）：Web MVP + 模型策略
目标
- 上线可用 MVP：用户输入 API Key + PDF/LaTeX + theorem，返回形式化进度与结果。
- 验证“用户自带 API Key”方案的可行性与风险边界。

任务
- React 前端：任务创建、运行状态、日志与结果展示。
- 后端编排：多模型路由、重试策略、失败回退策略。
- API Key 风险研究：封禁风险、速率限制、合规边界、告警机制。
- 集成多模型选择器：给用户可控的质量/速度/cost 档位。

交付物
- Web MVP（可实际提交任务并查看结果）。
- 风险评估文档（尤其是 Claude API key by user 模式）。
- 模型选择建议（默认策略 + 高级策略）。

验收
- 端到端流程稳定运行。
- 用户可独立完成一次从输入到结果的闭环。

第三阶段（2026-08-01 ~ 2026-12-31）：数学专用模型研究 + 自动证明
目标
- 验证数学微调/蒸馏模型在该任务上的性价比。
- 推进“自动证明新定理”的可行路径。

任务
- 调研并接入数学微调模型（如可用的 Archimedes 类模型）。
- 做 small model vs open-source flagship 对比实验。
- 建立自动证明范式：目标分解、子引理调度、失败恢复。
- 评估完全自动证明新定理的成功条件与失败模式。

交付物
- 研究报告（微调/蒸馏是否值得、收益边界）。
- 自动证明原型（针对限定类型定理）。
- 后续研究路线图（可投稿或继续迭代方向）。

验收
- 在限定题型上实现可复现的自动证明成功案例。
- 明确下一年度研究重点（扩题型 or 强化基础设施）。
